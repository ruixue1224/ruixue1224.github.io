一、Nginx 概述

1.1 什么是 Nginx

Nginx（Engine X）是一款轻量级、高性能的 HTTP 和反向代理服务器，同时也是一个 IMAP/POP3/SMTP 代理服务器。它由俄罗斯程序员 Igor Sysoev 于 2004 年首次发布，最初是为了解决 C10K 问题（即同时处理 10000 个客户端连接）而设计。如今，Nginx 已成为全球使用最广泛的 Web 服务器之一，根据 W3Techs 的统计，全球超过三分之一的网站在使用 Nginx，包括 Netflix、GitHub、WordPress.com 等高流量站点。

Nginx 的核心优势在于其事件驱动的异步非阻塞架构，这使得它在处理高并发连接时具有极低的内存消耗和极高的吞吐量。与传统基于线程或进程的 Web 服务器（如 Apache 的 prefork 模式）不同，Nginx 采用单线程事件循环模型，一个 Worker 进程即可处理数千个并发连接，极大地降低了上下文切换和内存开销。

1.2 Nginx 的核心功能

Nginx 的功能非常丰富，主要涵盖以下几个核心领域：

功能 说明
静态资源服务 高效处理 HTML、CSS、JS、图片等静态文件，性能远超传统 Web 服务器
反向代理 作为客户端与后端服务器之间的中介，隐藏真实服务器，实现请求转发和安全控制
负载均衡 将客户端请求分发到多个后端服务器，避免单点故障，提升系统吞吐量和可用性
HTTPS/SSL 支持 SSL/TLS 加密通信，保障数据传输安全
缓存 代理缓存后端响应，减少重复请求，降低后端压力
限流 控制请求速率，防止恶意攻击和过载
虚拟主机 支持基于域名或 IP 的虚拟主机配置，一台服务器可托管多个站点

1.3 Nginx vs Apache

对比项 Nginx Apache
架构模型 事件驱动、异步非阻塞 进程/线程驱动（prefork/worker）
并发处理 极强，单进程可处理数万连接 一般，每个连接占用一个线程
内存消耗 极低 较高
静态文件性能 极优 一般
动态内容处理 需反向代理至后端 可通过模块直接处理（如 mod_php）
配置灵活性 高，配置简洁 高，支持 .htaccess 分布式配置
模块机制 需编译时选择 支持动态加载模块
适用场景 高并发、反向代理、静态资源 动态内容丰富、需要 .htaccess 的场景

1.4 正向代理 vs 反向代理

理解正向代理和反向代理的区别是学习 Nginx 的重要基础：

  • 正向代理：代理的是客户端。客户端明确知道要访问的目标服务器，但通过代理服务器发送请求。典型场景：VPN、科学上网。客户端需要配置代理地址。
  • 反向代理：代理的是服务端。客户端不知道真实的服务器是谁，请求发送到代理服务器，由代理服务器转发到后端。典型场景：Nginx 反向代理后端的 Tomcat/Node.js 服务。客户端无需任何配置。

二、Nginx 架构原理

2.1 进程模型

Nginx 采用经典的 Master-Worker 多进程架构，这是其高性能和高可靠性的基石：

Master 进程的职责：

  • 读取并验证配置文件 nginx.conf
  • 创建、绑定和关闭 Socket
  • 管理 Worker 进程的创建、退出和重启
  • 接收并处理来自管理员的信号（如 reload、stop）
  • 不处理具体的业务请求

Worker 进程的职责：

  • 接受客户端的连接请求
  • 处理 HTTP 请求并返回响应
  • 与后端服务器通信（反向代理场景）
  • 各 Worker 进程之间相互独立，一个 Worker 崩溃不会影响其他 Worker

此外，Nginx 还有 Cache Manager 和 Cache Loader 进程，负责缓存的管理和加载。

2.2 事件驱动模型

Nginx 的高性能核心在于其事件驱动架构。每个 Worker 进程内部运行一个事件循环（Event Loop），基于操作系统的多路复用机制处理网络 I/O：

  • Linux：使用 epoll（默认且推荐）
  • FreeBSD/macOS：使用 kqueue
  • 其他 Unix：使用 select 或 poll（性能较差）

事件驱动的工作流程如下：

  1. Worker 进程在事件循环中等待事件（新连接、数据到达、定时器超时等）
  2. 当事件发生时，操作系统通知 Worker 进程
  3. Worker 进程处理该事件（非阻塞方式读取数据、发送响应等）
  4. 如果某个操作需要等待（如后端响应），Worker 不会阻塞，而是注册一个回调，继续处理其他事件
  5. 当等待的操作完成时，再次触发事件，Worker 继续处理

这种模型使得单个 Worker 进程可以同时处理数千甚至数万个并发连接，而不会因为某个慢请求阻塞其他请求。

7.4 if 指令

if 指令可以在 Server 和 Location 块中使用，但需谨慎，Nginx 官方称之为 “Evil”：

if 的判断条件：

  • = / !=：相等/不相等
  • ~ / ~*：正则匹配（区分/不区分大小写）
  • !~ / !~*：正则不匹配
  • -f / !-f：文件是否存在/不存在
  • -d / !-d：目录是否存在/不存在
  • -e / !-e：文件或目录是否存在/不存在
  • -x / !-x：文件是否可执行

注意：Nginx 官方建议尽量避免使用 if，推荐使用 try_files 或 map 替代。if 在 Location 中可能导致不可预期的行为，这就是所谓的 “if is evil” 问题。

九、负载均衡

负载均衡是 Nginx 反向代理的重要扩展，将请求分发到多个后端服务器，提升系统的吞吐量和高可用性。

9.1 Upstream 基本配置

9.2 负载均衡策略

Nginx 支持多种负载均衡算法：

1. 轮询（Round Robin）— 默认

每个请求按时间顺序逐一分配到不同的后端服务器，适合后端服务器性能相近的场景。

2. 加权轮询（Weighted Round Robin）

权重越高，分配到的请求越多。适合后端服务器性能不均的场景。

3. IP 哈希（IP Hash）

根据客户端 IP 的哈希值分配请求，同一 IP 的请求始终分配到同一后端服务器。适合需要会话保持（Session Sticky）的场景。

4. 最少连接（Least Connections）

将请求分配给当前活跃连接数最少的服务器。适合请求处理时间差异较大的场景。

5. 一致性哈希（Consistent Hash）

根据请求 URI 的一致性哈希分配请求，适合缓存场景，减少缓存失效。

6. 随机（Random）

随机选择两个服务器，然后从中选择活跃连接数较少的那个。适合分布式环境。

十八、常见故障排查

18.1 502 Bad Gateway

原因：Nginx 无法从上游服务器获取有效响应。

常见原因及解决方案：

原因 解决方案
后端服务未启动 检查后端服务状态，启动服务
后端服务端口错误 检查 proxy_pass 端口是否正确
后端服务进程崩溃 查看后端服务日志，修复崩溃原因
PHP-FPM 进程数不足 增加 pm.max_children
后端连接超时 增大 proxy_connect_timeout
防火墙阻止连接 检查防火墙规则
SELinux 限制 检查 SELinux 策略

18.2 504 Gateway Timeout

原因：Nginx 等待上游服务器响应超时。

常见原因及解决方案：

原因 解决方案
后端处理时间过长 优化后端代码，增加数据库索引等
proxy_read_timeout 过小 增大超时时间
后端服务器负载过高 扩容或优化后端服务
网络延迟过高 检查网络状况

18.3 413 Request Entity Too Large

原因：请求体超过 Nginx 允许的最大值。

18.4 499 Client Closed Request

原因：客户端在 Nginx 返回响应前关闭了连接（通常是客户端超时或主动取消请求）。

十九、Nginx 与 Lua / OpenResty

19.1 OpenResty 简介

OpenResty 是一个基于 Nginx 与 LuaJIT 的高性能 Web 平台，内部集成了大量精良的 Lua 库和第三方模块。它将 Nginx 变成一个强大的 Web 应用服务器，使得开发者可以使用 Lua 脚本快速开发高性能的 Web 应用、API 网关和微服务。

OpenResty 的核心优势：

  • 高性能：Lua 代码在 Nginx Worker 进程中直接执行，无需额外的进程间通信
  • 灵活性：可以在请求处理的各个阶段插入 Lua 代码
  • 生态丰富：内置了大量 Lua 库（如 JSON 解析、数据库驱动、HTTP 客户端等）
  • 非阻塞：所有 I/O 操作都是非阻塞的，不会阻塞 Worker 进程
